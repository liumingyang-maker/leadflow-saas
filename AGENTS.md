# AGENTS.md

本文件是 Codex、Claude Code 及其他编码代理在本仓库工作的最高级项目规则。开始任何任务前必须完整阅读。

## 1. 项目目标

LeadFlow 是面向中小外贸团队的线索发现与跟进工作台。核心流程：

```text
产品画像 → 搜索/导入 → 清洗去重 → 人工审核 → CRM → 个性化触达 → 回复/转化统计
```

优先保证数据隔离、安全、可恢复和核心闭环，不追求渠道数量。

## 2. 开工前必须阅读

按顺序：

1. `README_START_HERE.md`
2. `PRODUCT_SCOPE.md`
3. `ARCHITECTURE.md`
4. `UI_SYSTEM.md`
5. `TASKS.md`
6. 当前任务涉及目录中的 README 或 ADR

不得在未找到明确任务 ID 的情况下开始大规模修改。

## 3. 不可违反的架构规则

1. 新业务代码不得继续写入 `web/app.py`。
2. 路由层不得包含长业务流程、SQL 或第三方 API 调用。
3. Service 不得依赖 Flask `request/session/g/flash`。
4. Repository 的租户数据方法必须显式接收 `tenant_id`。
5. Integration 不得直接写业务数据库。
6. Job 只能调用 Service，不得复制路由逻辑。
7. 不得在模块 import 时启动线程、定时器或网络请求。
8. 不得在模板中实现套餐权限或业务规则。
9. 不得使用固定默认密码、示例密钥或生产假数据。
10. 不得无测试地修改认证、租户、支付、任务或迁移逻辑。

## 4. 安全规则

- 永远不要提交 `.env`、密钥、cookie、真实客户数据；
- 永远不要在日志打印密码、SMTP 授权码、API Key、完整 token；
- 所有浏览器会话写请求默认有 CSRF；
- 公开接口的豁免必须写明理由和替代保护；
- 所有对象读取必须验证租户所有权；
- 文件路径不得直接使用用户输入；
- 跳转 URL 必须来自服务端记录或通过签名；
- 支付回调必须验签、验金额、幂等；
- SQL 必须参数化；
- 上传必须限制类型、大小和文件名；
- 新增依赖前检查维护状态、许可证和安全风险。

发现安全问题时，停止扩展功能，优先提交最小安全修复。

## 5. AI 修改范围

一个任务：

- 只解决一个任务 ID；
- 默认不超过 8 个生产代码文件；
- 默认不超过 600 行净新增/修改；
- 不做无关重命名；
- 不顺手格式化整个仓库；
- 不升级无关依赖；
- 不删除旧兼容路径，除非任务明确允许；
- 超出范围时拆分新任务，不自行扩大。

## 6. 工作方式

### 开始前

输出：

```text
Task:
Goal:
Files expected:
Risks:
Tests planned:
```

先查看相关代码和测试，再修改。

### 实现中

- 小步提交；
- 先写或补回归测试；
- 使用现有风格，除非任务专门迁移风格；
- 将外部依赖放在接口后；
- 错误使用明确类型，不用大范围 `except Exception` 吞掉问题；
- 对用户可恢复错误给出稳定错误码；
- 所有时间使用 UTC 存储，展示时按用户时区转换。

### 完成后

必须输出：

```text
Summary:
Changed files:
Behavior changes:
Security impact:
Data/migration impact:
Tests run:
Manual checks:
Rollback:
Known limitations:
```

## 7. 测试要求

至少执行：

```bash
make lint
make test
make check
```

任务涉及 UI 时增加：

- 桌面核心流程；
- 375px 移动视图；
- 空/加载/错误状态。

任务涉及迁移时增加：

- 空库升级；
- 旧库升级；
- 重复执行；
- 迁移后数据校验；
- 备份恢复演练。

任务涉及租户时增加：

- tenant A 正常访问；
- tenant B 不能访问 A；
- 无 session；
- 暂停或过期；
- 管理员与普通用户边界。

不得通过删除测试、降低断言、全局 skip 或捕获异常来让 CI 变绿。

## 8. 代码质量

- Python 使用 Ruff 格式化与检查；
- 新公共函数写类型标注；
- 函数尽量小于 50 行；
- 路由尽量小于 30 行；
- 一个模块只承担一个业务目的；
- 避免布尔参数控制多个完全不同流程；
- 使用 dataclass/TypedDict/Pydantic 风格 DTO，避免随意 dict 跨层传递；
- 复杂决策写在 Domain/Policy，并有表格测试；
- 注释解释“为什么”，不复述代码。

## 9. 依赖规则

新增依赖必须在 PR 说明：

- 用途；
- 为什么标准库或现有依赖不够；
- 许可证；
- 维护状态；
- 是否影响镜像体积；
- 如何移除/回滚。

未经任务批准，不引入 React、Next.js、Celery、Kubernetes 或微服务框架。

## 10. 数据与迁移

- 任何 schema 变化必须有迁移；
- 不在应用启动时静默吞掉迁移失败；
- 迁移前先备份；
- 不可逆迁移需明确标识；
- 禁止直接删除客户数据；
- 删除走软删除或明确二次确认；
- 测试 fixture 不得包含真实客户数据。

## 11. UI 规则

- 使用 `UI_SYSTEM.md` token 和组件；
- 不在页面内复制大段 CSS；
- 文案不得使用未经证实的数字宣传；
- 第三方额度与能力来自 catalog/config；
- 按钮和状态命名保持统一；
- 所有异步任务使用统一 JobProgress；
- 所有危险操作使用 ConfirmDialog。

## 12. Git 和 PR

分支：

```text
task/P0-001-remove-default-admin
fix/P0-004-signed-click-redirect
refactor/P1-012-auth-blueprint
```

提交格式：

```text
type(scope): summary

Task: P0-001
```

PR 必须关联任务 ID，不允许一个 PR 混合多个独立任务。

## 13. 禁止项

- “顺便重构整个项目”；
- 将所有代码一次性搬目录；
- 大段复制网络代码而不核对许可证；
- 用 mock 数据伪装真实采集成功；
- 在生产失败时自动返回编造结果；
- 直接将第三方响应暴露给浏览器；
- 只凭页面能打开就宣布完成；
- 修改安全逻辑却不写负向测试；
- 在未备份的情况下跑破坏性迁移。
