# LeadFlow 仓库审查报告

## 1. 总体判断

### 当前成熟度

| 维度 | 评价 | 说明 |
|---|---|---|
| 产品原型 | 较强 | 已覆盖获客、清洗、评分、CRM、邮件、询盘、竞品监控、支付和后台 |
| 工程结构 | 偏弱 | 大量路由与业务逻辑集中于单文件 |
| 可测试性 | 偏弱 | 缺少清晰依赖边界，外部 API 与业务代码耦合 |
| 安全性 | 需要立即治理 | 默认管理员密码、CSRF、任务租户隔离、公开跳转等问题 |
| 可扩展性 | 偏弱 | 依赖单进程内存状态和后台线程 |
| UI 一致性 | 一般 | 页面数量多，但组件、文案和状态表达缺少统一规范 |
| AI 可协作性 | 偏弱 | 缺少任务边界、架构约束、测试门禁和变更模板 |

### 正确策略

**不推倒重写，采用绞杀式重构。**

保留现有用户可见行为和数据，逐个模块从 `web/app.py` 迁移到 Blueprint、Service、Repository、Integration 和 Job。每迁移一个模块就补测试，并保持旧 URL 可用。

## 2. 当前值得保留的资产

### 业务资产

- 面向中国外贸从业者的明确场景；
- 产品画像、关键词、目标国家等入驻流程；
- 多来源线索标准化与去重；
- 线索评分和 CRM 状态流转；
- 邮件模板、发送、打开/点击、退订和跟进；
- 独立站询盘接入；
- 多租户数据目录；
- 支付回调的验签、金额二次校验和幂等思路；
- SQLite 在线备份思路；
- BYOK 与平台额度两种配置思路。

### 技术资产

- Flask + Jinja 对当前团队规模足够简单；
- 现有模板可以渐进迁移，不必全部重做；
- 采集器已按文件拆分，具备进一步统一接口的基础；
- 租户数据分目录，MVP 阶段隔离直观；
- 密码已使用 PBKDF2，并兼容旧哈希升级；
- 退订使用签名 token，GET 确认、POST 执行的方向正确；
- 支付状态查询已校验订单所属租户。

## 3. 必须立即修改的 P0 问题

### P0-安全

1. **取消代码内自动创建固定管理员账号**
   - 当前逻辑会创建固定邮箱和固定密码。
   - 改为只允许通过环境变量或一次性 CLI 创建。
   - 首次登录强制改密码。
   - 生产启动时未配置管理员不得静默创建弱口令。

2. **全局 CSRF 防护**
   - 所有浏览器会话下的 POST/PUT/PATCH/DELETE 默认受保护。
   - 公开 webhook、支付通知、独立站询盘接口只做最小范围豁免，并使用签名、token、来源域名或幂等键保护。

3. **修复任务状态跨租户读取**
   - 当前任务状态只按 `task_id` 查询。
   - 任务记录必须包含 `tenant_id`。
   - 查询任务时必须校验当前租户。
   - 任务 ID 使用不可猜测 UUID，不能依赖时间前缀。

4. **修复点击追踪开放跳转**
   - 不能直接接受任意 `?u=https://...` 后跳转。
   - 发送邮件时存储目标 URL，点击时根据 tracking 记录读取。
   - 或对目标 URL 做带过期时间的签名。
   - 至少限制协议、最大长度和禁止危险主机/IP。

5. **Cookie 与代理配置**
   - 生产启用 `SESSION_COOKIE_SECURE=True`。
   - 明确 7 天或 60 天中的一个会话策略。
   - 只在可信反向代理后启用 `ProxyFix`。
   - 不能无条件相信客户端提交的 `X-Forwarded-For`。

6. **每次请求检查账号状态**
   - 不能只在登录时检查 suspended、trial expired、plan expired。
   - 在受保护请求前重新读取租户状态，或使用短 TTL 缓存。
   - 管理员暂停用户后，现有会话应尽快失效。

7. **密钥存储**
   - SMTP 密码、API Key 不应长期明文写入租户 JSON。
   - 第一阶段可使用 `FERNET_KEY` 加密敏感字段。
   - 第二阶段迁移到数据库密文列或云密钥管理服务。
   - 日志和异常不得打印密钥。

### P0-稳定性

1. **移除模块导入即启动的后台线程**
   - 自动跟进、竞品重扫、备份不能在导入 `app` 时启动。
   - Web 进程重启、扩容或调试导入都可能重复执行。
   - 使用独立 worker/scheduler；过渡期可用数据库任务表 + 单独进程。

2. **持久化任务状态**
   - `_task_status` 和内存限流重启后丢失。
   - 任务状态落库，记录 queued/running/succeeded/failed/cancelled。
   - 保存开始时间、结束时间、进度、错误摘要、租户、任务类型与参数摘要。

3. **数据库迁移版本化**
   - 停止用散落的 `ALTER TABLE + except` 作为长期迁移方案。
   - 引入 Alembic，或至少建立 `schema_migrations` 表和顺序迁移脚本。
   - 每个迁移必须可测试、可重复执行、有备份与回滚说明。

4. **外部集成统一超时和重试**
   - 每个 HTTP 请求必须有 connect/read timeout。
   - 只对安全、幂等的操作重试。
   - 记录 provider、耗时、状态和错误类别。
   - 失败不能让整个批次永久卡死。

## 4. 当前结构性问题

### 超大入口文件

`web/app.py` 同时承担：

- Flask 初始化；
- 会话和权限；
- 注册、登录、找回密码；
- 入驻；
- CRM；
- 邮件、WhatsApp；
- 询盘 API；
- 邮件追踪和退订；
- 支付；
- 采集；
- 评分和邮箱补充；
- 竞品雷达；
- 系统设置；
- 管理后台；
- 自动任务；
- 备份。

这会导致：

- AI 修改一个模块时读取上下文过大；
- 路由层直接操作数据库、配置和第三方 SDK；
- 难以模拟外部服务；
- 难以确定变更影响范围；
- 合并冲突频繁；
- 一个语法错误影响整个应用。

### 数据层边界不清

- 全局管理库和租户库有一定分离，但业务代码可直接取得路径和连接；
- 多个模块自行处理迁移；
- 租户隔离依赖调用者始终传入正确 `tid`；
- 缺少统一 Repository 约束；
- 配置 JSON、数据库和 session 各保存一部分状态，事实来源容易不一致。

### 集成接口不统一

不同采集器拥有不同字段、配置方式、返回格式和错误表达。应统一为：

```python
@dataclass(frozen=True)
class CollectionRequest:
    tenant_id: str
    product_profile: ProductProfile
    countries: list[str]
    limit: int
    options: dict[str, Any]

@dataclass
class CollectionResult:
    provider: str
    leads: list[LeadCandidate]
    warnings: list[str]
    usage: Usage
    raw_count: int
    duration_ms: int

class Collector(Protocol):
    key: str
    def validate_config(self, config: Mapping[str, Any]) -> list[str]: ...
    def collect(self, request: CollectionRequest) -> CollectionResult: ...
```

### UI、配置、帮助文案混在模板中

模板中同时放：

- 页面结构；
- 渠道营销文案；
- API Key 教程；
- 价格和免费额度；
- 风险提示；
- 可用性判断；
- 第三方平台事实。

第三方价格、额度和能力会变化，不能长期硬编码在模板里。应放进渠道注册表，例如 `integrations/catalog.yaml`，并由后台状态决定显示。

## 6. 建议的改造原则

1. **先加护栏，再搬代码。**
2. **每次只迁移一个业务域。**
3. **路由只做输入解析、权限、调用 Service、返回结果。**
4. **Service 不直接依赖 Flask 全局对象。**
5. **Repository 强制接受 `tenant_id`。**
6. **Integration 不直接写数据库。**
7. **Job 调用 Service，不能复制业务逻辑。**
8. **模板只展示，不决定套餐额度和业务权限。**
9. **所有用户可见状态都有唯一状态枚举。**
10. **没有测试的重构不合并。**

---

## 7. 最终判断

这个项目不差，问题是“功能密度很高，工程边界很低”。它适合继续做，但要从“让 AI 不断添加代码”切换为“让 AI 按架构合同迁移和验证代码”。

现阶段最有价值的工作不是再加五个渠道，而是让以下闭环稳定：

> 产品画像 → 搜索/导入 → 清洗去重 → 人工审核 → CRM → 个性化触达 → 回复/转化统计
